Python の関数は、始まりの def から 終わりの return まで、
ひとまとまりの「道具」として動きます。途中で勝手に動き出しません。

05 — コードで見る

def から return までを、コードで見比べる。

同じ「レンジ１台」の構造が、Python コードの中にもそのまま現れます。

warm_up.py
# ▼ ボタン作り（設計だけ。まだ動かない）
def 温め(numbers):
    # ▼ 加熱中（中でぐるぐる計算）
    合計 = 0
    for n in numbers:
        合計 = 合計 + n

    # ▼ チーン！と外へ届ける
    return 合計


# ── 呼び出し側 ──────────────
study_data = [10, 20, 30]
結果 = 温め(study_data)
DEF · 設計

def 温め(numbers): で「温め」という名前と、
中に入れる材料 numbers を決めている。

HEATING · 加熱中

for ループの中で、合計 をぐるぐる更新。
これがレンジ内部で「回っている」状態。

RETURN · 完成

return 合計 で、外（呼び出し側）に値を届ける。
受け取った側は 結果 という変数で持つ。

呼び出すときは study_data なのに、

study_data は、あなたが実際にレンジに入れようとしている「カレー（実物のデータ）」。
numbers は、レンジの設計図に書かれた「温めるもの（仮の名前）」。

07 — 外と中、ふたつの呼び名

同じ中身に、立場の違う２つの名札がついている。

レンジは、カレーでも肉まんでも動くように、あえて汎用的な仮名で待ち構えています。

OUTSIDE · 呼び出し側
study_data
あなたが レンジに入れようとしている、
実物の「カレー」。
CURRY
[10, 20, 30]
＝ 具体的な実物データ
▶
代入 / PASS
▶
INSIDE · 関数の中
numbers
設計図に書かれた 「温めるもの（食品）」 という、
関数の中だけの共通の呼び名。
FOOD ?
中身は何でも OK（カレー / 肉まん / ご飯…）
＝ 汎用的な仮の名前
つまり、こういう関係

関数という箱の中だけの「共通の呼び名 numbers」に 代入して処理している。

08 — まとめ

今日おぼえて帰る、関数の３つの顔。

関数は「ボタン作り → 加熱中 → チーン！」の３ステップ。
引数の名前は、外と中で違っていて当たり前。

STEP 01
defボタン作り
関数の名前と、入口（引数）を決めるだけ。
書いた瞬間は、まだ何も動いていない。
STEP 02
for加熱中
関数が呼び出されると、中で処理がぐるぐる回る。
ループ・条件分岐・計算はぜんぶこの中。
STEP 03
returnチーン！
結果を 外へ届ける ゴール。
return を書かないと、外には何も渡らない。
TAKEAWAY
「study_data」は外の実物、「numbers」は中の仮の呼び名。
レンジは中身を選ばないから、わざと汎用的な名前で待っている。
質問や不明点があればチャットで聞いてください。
